# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

MCP server that converts heavy Cocos Creator `.scene` files (~700KB JSON) into compact semantic scene graphs (~20KB) for AI coding assistants, and edits scenes/prefabs through semantic operations. Nineteen MCP tools: scene reading (`query_scene_graph`, `query_prefab_graph`, `query_animgraph`, `list_scene_scripts`, `find_scene_nodes` — by name pattern and/or component, `inspect_node`), project/asset introspection (`get_project_info`, `get_asset_info`, `list_assets`, `find_asset_references` — reverse "who uses this asset/UUID" lookup), writing (`apply_edits`, `validate_document`, `build_prefab`, `build_animgraph`, `create_asset_meta` — offline `.meta` generation so assets created outside the editor are importable with a stable UUID), and measurement/hygiene (`get_node_bounds`, `compute_fit_scale`, `compute_rotation`, `lint_assets`).

Locked decisions (details and verified Cocos 3.x format notes live in [README.md](README.md)): LLMs never text-edit `.scene`/`.prefab` files — only semantic operations; nodes are addressed by path or stable `_id`, never by `__id__`; edit the source prefab, not the instance (instance internals exist only as overrides); wrapper convention — a model/sprite is never the prefab root (`Root` → `Visual` child). A companion Claude Code SKILL `cocos-scene-builder` (policy/recipes on top of these tools) is distributed outside this repo.

## Commands

```bash
npm start                # Run MCP server (stdio transport)
npm test                 # Run all tests (Node.js built-in test runner)
npm run test:watch       # Watch mode
npm run test:coverage    # With coverage (--experimental-test-coverage)
node --test test/core/SceneParser.test.js  # Run a single test file
npm run cli -- <scene-path> [--json] [-o file]  # CLI without MCP
```

## Architecture

ES modules (`"type": "module"`), plain JavaScript, Node >= 18, single dependency (`@modelcontextprotocol/sdk`).

### Data Flow

```
.scene JSON → SceneParser (index) → NodeTreeBuilder (tree) → Formatter (output)
                                       ↑           ↑
                                  TypeFilter    ScriptResolver
                                  NodeFilter    PropertyExtractor
```

**SceneParser** reads the flat JSON array from a `.scene` file and indexes objects into nodes (`cc.Node`/`cc.Scene`) and components maps. **ScriptResolver** scans `assets/**/*.{ts,js}.meta` files and maps compressed component UUIDs to script names by exact compression (`src/utils/uuid.js`). **NodeTreeBuilder** recursively builds a minified tree, using **TypeFilter** to skip noise types (transforms, UI renderers), **NodeFilter** to skip skeleton bones, and **PropertyExtractor** to pull meaningful props from custom script components (and built-in `cc.*` in detailed mode); asset references — including `_materials` arrays — resolve to names via `AssetIndex.label()`: sub-assets as `file@subId`, fbx-baked materials marked `(embedded)`, engine built-ins as `builtin:<name>`. Formatters (Text/Json/Stats) render the final output.

**SceneMinifier** is the facade/orchestrator that wires all core classes together via constructor injection.

### Asset Layer

**AssetIndex** scans `assets/**/*.meta` and resolves any reference form — project-relative path, full UUID, compressed UUID, or `<uuid>@<subId>` sub-asset — plus filtered listing (type/folder/pattern). **builtins.js** resolves UUIDs of the editor's internal db (`db://internal`: primitives, default materials/effects/spriteFrames/fonts) against the static table `src/assets-data/builtins-3.8.json` (407 assets, harvested from an installed Cocos Creator 3.8.7 by `scripts/harvest-builtins.mjs`): `AssetIndex.label()` falls back to `builtin:<name>`, `get_asset_info` answers with an "engine built-in" card, and the validator's missing-asset warning excludes known builtins (only genuinely unknown UUIDs are reported). **FindAssetReferences** is the reverse lookup: a string scan over `assets/**` serialized files, then a deep pass over matching scenes/prefabs attributing every `__uuid__` hit to `node path ▸ component.property` (hits inside standalone value objects climb to their owner through a reverse-reference map); `.mtl`/`.anim`/`.animgraph` files report at file level. **AssetInspector** extracts per-type details: sprite rect/trim/9-slice from meta subMetas, mesh AABB from compiled `library/<xx>/<uuid>@<subId>.json` (fallback: glTF POSITION accessors for `.glb`), prefab summaries, material effect/defines. **ProjectInfoReader** reads engine version (`package.json` → `creator.version`), designResolution/layers/physics (`settings/v2/packages/`). `src/utils/uuid.js` implements the verified Cocos UUID compression algorithm (5 hex chars kept + 3-hex→2-base64 packing) and `nameToId` — subMeta ids are 5 chars sampled from the md5 of the sub-asset name (ported from npm `@cocos/asset-db`, verified: `texture`→`6c48a`, `spriteFrame`→`f9941`); `src/utils/glb.js` reads GLB JSON chunks.

### Write Layer

`src/document/` is the lossless counterpart to the lossy read pipeline. **SceneDocument** loads the flat JSON array, addresses nodes by path (`Canvas/Panel/BuyBtn`, `Name[i]` disambiguation) or scene `_id`, and saves atomically (temp+rename). Canonical form — verified byte-identical with editor output on the golden corpus — is depth-first first-visit numbering over the whole reference graph plus `JSON.stringify(arr, null, 2)` with no trailing newline; `renumber()` restores it and garbage-collects unreachable objects. **operations.js** implements the fifteen semantic ops (all-or-nothing batches); property paths navigate THROUGH `{__id__}` refs to standalone value objects (`cc.Line` `width.constant` edits the referenced `cc.CurveRange`; object values merge through the ref, primitive writes are rejected with a hint). `set_component_property` only edits the FIELDS of an existing element of a `CCClass[]` property (a `{__id__}`-ref array like `CardsBase.entries: CardEntry[]`) — changing the SET of elements is `insert_array_element`/`remove_array_element`: insert deep-clones a neighbor element as the skeleton (that is where the full field set comes from — user CCClasses have no default registry; `cloneOwnedElement` allocates FRESH owned sub-objects so `config`/`members` never alias the source), merges `value` over the top-level fields, and splices in the new `{__id__}`; remove splices the element out and, for a reference to an OWNED value object, GCs its subtree on renumber with the same reachability guard as `remove_node`/`remove_component` — a reference to a live node/component (`pipeControllers: Foo[]`) is just a dropped link, the target stays. Ownership is decided by `_components`/`MountedComponentsInfo` MEMBERSHIP (`componentIdxSet`/`isValueObjectIdx` in operations.js, mirrored in PropertyExtractor), never by the `node` back-ref heuristic — a user data struct with a `@property node: cc.Node` field is still an owned struct. Both splice ops remap `cc.TargetOverrideInfo.propertyPath` index segments running through the array (drop for the removed element, ±1 for shifted ones — `remapTargetOverridesForSplice`), and insert rejects a `from` pointing at a null hole (nothing to clone; an explicit reference value is still accepted). A `guardTypedRefArrayWrite` footgun check rejects a bare-value write into the append slot / a null hole of such a ref-array (which would otherwise serialize an inline object where the engine expects a `{__id__}` — silent corruption), pointing at `insert_array_element`; writing `null` is allowed (null holes are a legal serialization present in the golden corpus). Inline value-type arrays (`cc.Vec2[]`, any type in the shared `src/core/valueTypes.js` registry — single source for read rendering and write membership, incl. `cc.Mat4`) and scalar arrays are supported too. `prune_dangling_overrides` tidies editor-left dead `cc.TargetOverrideInfo` records (null/detached/invalid endpoints the engine skips on load; editor forms like `target: null` are never flagged). These dead records are NON-blocking: the validator reports each as a single warning (they cannot affect the runtime, and the editor regenerates some — e.g. a half-written `source: null` skeleton — on every save, so erroring would create an MCP↔editor whack-a-mole), so `apply_edits` proceeds without a prune call; the op stays available as opt-in cleanup and `validate_document` hints at it whenever such records exist. `overrideDeadReasons` (exported from targetOverrides.js) is the single source of truth for "engine skips this on load" — shared by the validator and prune, it subsumes the old structural error branches, so only sourceInfo↔source consistency on a LIVE record is still a hard error; `add_component` auto-adds missing companions BEFORE the component itself, mirroring the editor (adding `cc.Sprite`/`cc.Widget`/… inserts `cc.UITransform` first; `cc.SafeArea` pulls in `cc.Widget`→`cc.UITransform`; `cc.LabelOutline` pulls in `cc.Label`) — idempotent, so it never adds a second `cc.UITransform` to a node that already has one; the forward map `COMPANIONS_FOR` is derived from `REQUIRED_COMPANIONS`, and `attachCreatedComponent`/`ensureCompanions` share the template-instantiation path. Without this a UI node lacking `cc.UITransform` serialized as "valid" but crashed the editor on scene activation (`Widget.onEnable`/`align` read a null UITransform). `remove_component` mirrors `remove_node` (detach → reachability-filtered external-ref check → GC on renumber) with a `REQUIRED_COMPANIONS` guard (`cc.UITransform` is not removable while UI components need it, `force` does not bypass); on an instance stub it first tries the components MOUNTED on the instance (physical removal — an emptied `cc.MountedComponentsInfo` record goes entirely, NO removedComponents entry), otherwise delegates to instances.js (removedComponents). `set_component_property`/`set_asset_ref` on a stub address mounted components the same way (`findMountedComponent`: type/script-name matching, `target: "<mount path>"` / `componentIndex` disambiguation; a no-match error lists the mounted components and points source-prefab properties to `set_instance_property`). **targetOverrides.js** implements `@property` references INTO collapsed instances: `$node`/`$component` values that resolve inside a stub serialize as `null` plus a `cc.TargetOverrideInfo` in the document registry — `source` = plain component with `sourceInfo: null` (B1 form), or, when the source component itself lives inside an instance (`set_instance_property` with a reference value), `source` = that instance's stub with `sourceInfo` = TargetInfo carrying the component's fileId (B2 form, golden scene shape); always single-hop `localID`. Overwriting the property with a plain value drops the record (a live override shadows the serialized value on load). References from an instance to PLAIN scene objects are a different mechanism — a regular `{__id__}` inside the `CCPropertyOverrideInfo` value (golden: `playerGoldStorage`). A `{"$component": {node: "Stub", type}}` value without `target` prefers a component MOUNTED on the stub and stays a plain `{__id__}` (the golden scene references mounted components that way); mounted components as reference SOURCES use the plain B1 form. **ComponentTemplates.js** holds full 3.8.7 field sets taken from real files — ~44 templates covering UI/2D, 3D rendering, lights (with `cc.StaticLightSettings` extras), 3D+2D physics, audio and particles; field sets/order verified against editor-saved production projects, values are engine 3.8.7 defaults. Template placeholders: `{__ref__: i}` into extras (wired deeply — extras may reference other extras, and two keys may share one ref: `startSize`/`startSizeX` alias the same CurveRange), `{__self_node__}` (Button `_target`), `{__self_component__}` (TrailModule back-ref to its ParticleSystem). `PAIRED_FIELDS` in operations.js is keyed by the holder object's `__type__`, so getter/setter twins inside referenced module objects (`shapeType`/`_shapeType`, `numTilesX/Y`) and light `*HDR`/plain twins mirror on every write; `cc.Mask` needs a sibling `cc.Graphics` with `_fillColor` alpha 0 (companion the editor auto-adds, not extras). **Validator.js** checks referential/bidirectional invariants (errors) and conventions (warnings), including TargetOverrideInfo and MountedComponentsInfo/MountedChildrenInfo shapes (errors) and localID resolution against source prefabs (warnings, needs `{projectRoot}` option); the missing-asset warning excludes known engine builtins. **PrefabBuilder.js** compiles a compact spec into a prefab by generating an ops batch over a skeleton document; the wrapper convention (`visual` → child node) is built in. **MetaGenerator.js** generates `.meta` files offline (`create_asset_meta` tool; also backs the builders' `createMeta` and folder metas in the build tools) — the editor trusts a brought-along meta and keeps its UUID on import, so agent-created scripts/`.mtl`/`.anim`/images/audio/folders are addressable without opening the editor; `ver` values pinned to the 3.8.7 importer table (app.asar engine-extends contributions, newer editors migrate forward), image metas get full editor-shaped subMetas (ids via `nameToId`, sizes from PNG/JPEG headers, trim as full-image placeholders with `trimType: "auto"` the importer recomputes), idempotent (existing metas are never touched — no UUID churn), models (`.fbx/.glb/.gltf`) are deliberately refused: their userData derives from model content and the MCP itself needs the editor-produced `library/` artifacts. **MiniTree.js** renders compact subtrees for `apply_edits` responses.

### Animation Graph Layer

A `.animgraph` is the same flat JSON array with head `cc.animation.AnimationGraph` — `SceneDocument` serves as its plain array container (load/renumber/serialize/save apply; `root`/`resolveNode` don't), round-trip verified byte-identical on the golden Player.animgraph including the `_variables` object-map. **AnimGraphReader.js** parses the array into a semantic model (variables, per-layer states/transitions, conditions as strings) and renders the compact text summary; enum mappings (BinaryOperator `== != < <= > >=` = 0–5; VariableType float 0 / boolean 1 / trigger 2 / integer 3) are verified against the 3.8.7 engine sources. **AnimGraphBuilder.js** compiles a compact spec (`when: "Speed > 0.1"` strings, `trigger` keys) into editor-shaped objects — Entry/Exit/Any and the event-binding pairs (Motion in/out, AnimationTransition start/end) are always generated, objects are appended in canonical DFS order so `renumber()` is an identity, plus a `.meta` (`importer: "animation-graph"`). Tools: `query_animgraph` (path or UUID) and `build_animgraph`. The `cc.animation.AnimationController` template works with the `PAIRED_FIELDS` rule in operations.js: the graph reference serializes twice (`_graph` + `graph`), and every write via `set_asset_ref`/`set_component_property` mirrors both fields (deep-copied, never aliased). The controller belongs on the skeleton root inside a model — it cannot land on a collapsed instance (unpacking the model in the editor stays manual).

### Instance & Measurement Layer

**instances.js** implements collapsed prefab instances exactly as the editor serializes them: `instantiate_prefab` creates a stub node + `cc.PrefabInfo` (fileId = source prefab root's fileId) + `cc.PrefabInstance`, registers it in the document's instance registry, and accepts `.prefab` assets or model files (gltf-scene prefab read from `library/`); `set_instance_property`/`remove_instance_override` manage `CCPropertyOverrideInfo` objects, resolving `target` paths inside the source prefab into `TargetInfo.localID` fileIds (node PrefabInfo / component CompPrefabInfo); `remove_component` on a stub records a `cc.TargetInfo {localID: [CompPrefabInfo.fileId]}` in `cc.PrefabInstance.removedComponents` (the form the 3.8.7 engine reads in `applyRemovedComponents`; editor re-save verification pending) with dedup, cleanup of the dying component's overrides and an incoming-targetOverrides guard — `restore_instance_component` undoes it. Instance internals are never editable directly — only via overrides; multi-hop targets (inside nested instances) are not implemented. **Mounted components/children** (objects added ON TOP of an instance; golden shape: `cc.MountedComponentsInfo {targetInfo → localID: [mount-node fileId], components: [{__id__}]}`, components are regular serialized objects of THIS file with `node` → stub, `__prefab: null`, `__editorExtras__.mountedRoot` → stub; `MountedChildrenInfo.nodes` are regular `cc.Node`s with `_parent: null`) are readable AND editable: `mountedComponentEntries`/`mountedChildrenEntries`/`listMountedComponents`/`findMountedComponent` in instances.js provide the addressing, `inspect_node` renders them as **Mounted components / Mounted children** sections (full properties via PropertyExtractor with a `refResolver` that emits node/component addresses), `find_scene_nodes {component}` matches them, and the ordinary component ops edit/remove them (see Write Layer). Creating NEW mounted records via MCP is deliberately not implemented (editor work); editor re-save verification of mounted-component writes is pending. Reading instance internals: `inspect_node` on a stub expands the source prefab internals (read-only tree with `target` paths + the override list resolved from fileIds) plus an "Incoming scene references" section (targetOverrides pointing at this stub), and the `materials` check of `lint_assets` walks instances the same way to flag embedded-vs-project material mismatches per mesh. On the read side, **PropertyExtractor** surfaces target-overridden properties (serialized `null`) as sibling `<prop>__targetOverride` entries — additive only, so main-branch MCP clients keep working (locked by the snapshot tests in `test/tools/readSnapshots.test.js`; regenerate deliberately with `UPDATE_SNAPSHOTS=1`, diffs must be added lines/fields only). **Bounds.js** computes subtree AABBs (mesh AABBs via AssetInspector, UITransform rects, instance stubs through their source prefab with overrides applied) in two frames: `local` (node's own frame — BoxCollider center/size) and `world`; matrix math lives in `src/utils/math3d.js`, which also holds the verified quaternion ports (`eulerToQuat` = `cc.Quat.fromEuler`, `quatToEulerYZX` = `cc.Quat.toEuler`, round-trip corpus-verified on every `_lrot` of the golden and real scenes) behind the `compute_rotation` tool — quat↔euler conversion, axis composition in world/local space (parent-chain compensation when given `filePath`+`node`; stubs read their `_lrot` override), and orient ("local axis A looks along world direction D", optional `up` roll), always answering with euler+quat+a ready-to-apply op.

Never edit `.scene`/`.prefab`/`.animgraph` files with text edits — always go through `apply_edits`/`build_prefab`/`build_animgraph`.

### MCP Tools

Each tool extends `BaseTool` (abstract with `name`, `description`, `inputSchema`, `execute`; optional `aliases` maps courtesy argument keys onto canonical ones, e.g. `filePath`→`scenePath`). The server calls `BaseTool.run()`, which normalizes aliases and validates args against `inputSchema` (`src/tools/validateArgs.js`: unknown-key rejection with a did-you-mean hint, required/type/enum checks) before delegating to `execute()` — a bad call gets a self-explanatory error, never an internal crash like `paths[1] ... undefined`. Tools are registered in `src/tools/index.js:createTools()` and receive `projectRoot` from the server. The server (`McpServer`) uses `@modelcontextprotocol/sdk` with stdio transport.

### Key Convention: COCOS_PROJECT_ROOT

The MCP server resolves scene paths relative to `COCOS_PROJECT_ROOT` env var (falls back to `cwd()`). This is the root of the Cocos Creator project being analyzed, not this repo.

## Testing

Tests use Node.js built-in `node:test` and `node:assert`. Test fixtures live in `test/fixtures/` with a sample scene JSON, mock script `.meta` files (`test/fixtures/assets/`), and `mock-project/` — a miniature Cocos project (settings, asset metas copied from a real project, `library/` mesh + compiled gltf-scene prefab cache, generated `.glb`, prefabs for instance/bounds tests, deliberately bad-named assets for lint tests) used by asset/write-layer tests. `test/fixtures/golden/` holds unmodified editor-saved files from the real game (1.2MB scene + 4 prefabs + Player.animgraph): the write layer's round-trip tests must reproduce them byte-for-byte, so never reformat or hand-edit them. Test file structure mirrors `src/` (e.g., `test/core/SceneParser.test.js` tests `src/core/SceneParser.js`).

A real production Cocos Creator 3.8.7 3D project lives in `project-example/zombie-miner` (gitignored) — use it for manual verification of new tools against real data.

## Adding a New MCP Tool

1. Create `src/tools/NewTool.js` extending `BaseTool`
2. Implement `name`, `description`, `inputSchema` getters and `execute(args, projectRoot)`
3. Add to `createTools()` array in `src/tools/index.js`
4. Add test in `test/tools/NewTool.test.js`
