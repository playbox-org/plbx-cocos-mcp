# Анализ: добавление и удаление элементов массивов-свойств (CCClass[])

Причинно-следственный разбор того, почему write-слой умеет **редактировать поля**
существующих элементов массива вроде `CardsBase.entries: CardEntry[]`, но **не умеет
добавлять/удалять сами элементы** — и что для этого нужно.

Контекст: обнаружено при работе с `project-example/supercell-clashroyale`
(`CardsBase.entries` — массив вложенных `@ccclass CardEntry`, каждый с `type`,
`prefab`, `config`, `offsets`, `members`). Read-слой развёрнут отдельным фиксом в
`PropertyExtractor` (см. `__struct__`-разворачивание); этот документ — про write.

---

## 1. Как это устроено в файле (исходный факт)

`@property([CardEntry]) entries` сериализуется как **массив ссылок**, а не как массив
инлайн-объектов:

```json
// компонент CardsBase
"entries": [ { "__id__": 511 }, { "__id__": 512 }, ... ]

// объект 511 — отдельная запись в плоском массиве .scene
{ "__type__": "CardEntry",
  "type": 1,
  "prefab": { "__uuid__": "98238…", "__expectedType__": "cc.Prefab" },
  "config": { "__id__": 512 },              // ещё один вложенный CCClass по ссылке
  "offsets": [ {"__type__":"cc.Vec2","x":-20,"y":0}, … ],  // инлайн value-типы
  "members": [] }
```

Ключевые свойства формата, из которых вытекает всё остальное:

- **F1.** Элемент такого массива — это `{__id__:N}` → **standalone-объект** с собственным
  `__type__` в плоском массиве. Инлайн-объект без `__type__` там нелегитимен.
- **F2.** Полный набор полей типа `CardEntry` (и их дефолты) **в сцене не записан** — есть
  только фактические значения уже существующих элементов. Определение типа
  (`@property` декораторы) живёт в TS-исходнике/метаданных скрипта, не в `.scene`.
- **F3.** Канонический порядок объектов восстанавливается `doc.renumber()` обходом графа
  ссылок в глубину; недостижимые объекты при этом **удаляются сборщиком** (GC).

---

## 2. Что РАБОТАЕТ: правка полей существующих элементов

`set_component_property` / `set_asset_ref` по dotted-пути редактируют поля уже
существующих элементов. Проверено на реальной сцене:

| Путь | Что редактирует | Итог |
|---|---|---|
| `entries.0.type` | enum int на вложенном `CardEntry` | ✅ |
| `entries.0.offsets.0.x` | инлайн `cc.Vec2` внутри структуры | ✅ |
| `entries.0.prefab` (`set_asset_ref`) | asset-ref на структуре | ✅ |
| `entries.0.config.icon` (`set_asset_ref`) | asset-ref на структуре двумя уровнями глубже | ✅ |

**Причинная цепочка (почему работает):**

1. `parsePropertyPath` ([operations.js:213](../src/document/operations.js#L213)) понимает
   и `entries.0.type`, и `entries[0].type` — числовой сегмент индексирует массив.
2. `locateProperty` ([operations.js:248](../src/document/operations.js#L248)) при
   навигации встречает `{__id__}` и **следует по ссылке** — но только если целевой
   объект «value object»: `ref.__type__ && !doc.isNode(ref) && !isRef(ref.node)`
   ([operations.js:277](../src/document/operations.js#L277)).
3. `CardEntry` / `CardConfig` под это подходят: у них есть `__type__`, они **не ноды** и у
   них **нет back-ref `node`** (это то, что отличает data-CCClass от компонента).
   ⇒ навигация проходит насквозь в стандалон-объект и правит его поле.

Тот же предикат используется в read-фиксе `PropertyExtractor.#isDataStruct` — поэтому
read и write согласованы: что `inspect_node` показал как `__struct__`, то `apply_edits`
может отредактировать по тому же пути.

## 3. Что НЕ работает: изменение НАБОРА элементов

Среди 13 семантических операций
(`HANDLERS`, [operations.js:107](../src/document/operations.js#L107)) **нет** операции
«добавить/удалить элемент произвольного массива-свойства». Создание пары
«standalone-объект + ссылка» реализовано только для конкретных структур: `add_node`
(нода + `_children`) и `add_component` (компонент + `_components`).

### 3a. «Добавление» через запись индекса ПОРТИТ файл

`set_component_property property:"entries.10" value:{type:99}` формально проходит, но
записывает **голый инлайн-объект** `{"type":99}` вместо `{__id__}` → standalone `CardEntry`.

**Причинная цепочка (почему портит):**

1. Проверка границ ([operations.js:298](../src/document/operations.js#L298)) отбрасывает
   только `key > length`. Для `entries.10` при длине 10 условие `10 > 10` ложно ⇒ запись в
   позицию-append разрешена.
2. Запись — это **листовое присваивание** значения
   ([setPropertyOnComponent:1091-1093](../src/document/operations.js#L1091)):
   `container[key] = isReference ? value : mergeTyped(existing, value)`.
3. `existing = container[10] = undefined` ⇒ `isRef(undefined) = false` ⇒ ветка «write
   THROUGH ref» пропускается (нет ссылки, сквозь которую сливать).
4. `mergeTyped(undefined, {type:99})` при `existing` без `__type__` возвращает `given`
   как есть ([mergeTyped:179](../src/document/operations.js#L179)) ⇒ в массив ложится
   `{type:99}`.

**Следствие:** нарушен инвариант **F1** — элемент не является `{__id__}`-ссылкой на
типизированный объект; при загрузке движком это не десериализуется как `CardEntry`
(нет `__type__`, нет обязательных полей). Тихая порча сцены, `apply_edits` при этом
рапортует успех.

> Контраст: `set_component_property property:"entries.0" value:{type:3}` (индекс
> существует) — **безопасно** сливается в существующий `CardEntry` сквозь ссылку
> (ветка `isValueObject`, [operations.js:1072-1090](../src/document/operations.js#L1072)):
> `offsets` и прочие поля сохраняются, ссылка не заменяется. Разница именно в том, есть
> ли по индексу существующая `{__id__}`-ссылка.

### 3b. Удаление одного элемента — операции нет

Отдельной операции «убрать `entries[i]`» не существует. Можно затереть весь массив
(`entries = []`) — осиротевшие `CardEntry` станут недостижимы и уберутся GC при
renumber (**F3**), — но выразить «оставить рефы 0,2,3, убрать 1» через generic-запись
корректно нельзя (нужно вручную собрать массив `{__id__}` уже с новой нумерацией, а
она назначается только на save).

---

## 4. Корневые причины (архитектурный уровень)

- **R1. `set_component_property` — писатель ЛИСТОВЫХ значений, а не структуры массива.**
  Он не знает, что данный массив хранит ссылки на типизированные standalone-объекты, и
  что новый элемент требует *аллокации* объекта нужного типа + ссылки. Отсюда 3a.

- **R2. Тип элемента и его дефолты в сцене отсутствуют (F2).** Чтобы создать валидный
  `CardEntry`, нужен его полный набор полей. Для `cc.*` это решено каталогом
  `ComponentTemplates.js`, но для **пользовательских** CCClass (`CardEntry`, `CardConfig`,
  `CardMember`, …) реестра шаблонов нет и вывести его из частичного значения нельзя
  безопасно. Тип можно *подсмотреть* у соседнего существующего элемента
  (`entries[0].__type__` + его ключи), но это эвристика, а не источник истины.

- **R3. `mergeTyped` намеренно без `doc` и без создания ссылок**
  ([operations.js:176-211](../src/document/operations.js#L176)). Это by design: чистый
  слияльщик значений, который *не* умеет ходить по `{__id__}` и *не* создаёт рефы — чтобы
  случайно не осиротить объект и не заменить ссылку на голый объект. Поэтому «умное»
  добавление не может «просто получиться» внутри существующего пути записи.

- **R4. Машинерия аллокации/GC уже есть.** `doc.addObject()` добавляет объект и возвращает
  индекс; `add_node`/`add_component` этим пользуются; `renumber()` восстанавливает
  канонический порядок и собирает мусор. То есть **удаление — более простая половина**
  (splice ссылки + авто-GC), а **добавление — сложная** (нужны R2: тип + дефолты).

Итог: невозможность — не про «формат не позволяет», а про то, что **обобщённая операция
над массивом CCClass-ссылок ещё не написана**, а generic-запись значением для этого
структурно не предназначена (R1) и не имеет данных о типе (R2).

## 5. Что нужно для поддержки (эскиз дизайна)

Две новые семантические операции по образцу `add_node`/`remove_node`:

### `insert_array_element`
```
{ op: "insert_array_element",
  node, component, property: "entries",   // путь до массива через locateProperty
  index?: <int>,                           // undefined = append
  type?: "CardEntry",                      // если опущен — взять __type__ соседнего элемента
  value?: { type: 3, ... } }               // частичные поля поверх дефолтов
```
Механика:
1. `locateProperty` до массива-контейнера (уже умеет следовать по рефам к нужному месту).
2. Определить `__type__` элемента: явный `type`, иначе `__type__` существующего соседа;
   если массив пуст и `type` не задан — **ошибка** (R2: тип неоткуда взять).
3. Собрать каркас объекта нужного типа (по существующему соседу как шаблону, либо по
   явно переданным полям), пропустить `value` через `mergeTyped`.
4. `doc.addObject(obj)` → `idx`; `container.splice(at, 0, {__id__: idx})`.
5. Вложенные ссылочные поля (`config` и т.п.) — рекурсивно так же (или требовать их явного
   задания); осторожно с обязательными не-null рефами.
6. `renumber()` на save восстановит канонический порядок.

### `remove_array_element`
```
{ op: "remove_array_element", node, component, property: "entries", index: <int> }
```
Механика: `container.splice(index, 1)` — убрать `{__id__}`; осиротевший standalone-объект
(и его под-объекты, напр. `config`) станут недостижимы и уйдут GC при renumber (F3).
Проверить, что на удаляемый объект нет внешних ссылок (по образцу reachability-guard в
`remove_node`/`remove_component`).

### Обязательный guard в существующем коде
Закрыть footgun 3a: в `setPropertyOnComponent`/`setAssetRef` при записи по числовому ключу
в **позицию за концом массива** ссылок (когда соседние элементы — `{__id__}` на объекты с
`__type__`) — не писать голый объект, а **отклонять** с подсказкой использовать
`insert_array_element`. Это чистое дополнение, ломать существующее поведение не должно.

## 6. Крайние случаи / риски

- **Пустой массив + неизвестный тип** — добавление невозможно без явного `type` (R2).
- **Обязательные не-null ссылки внутри элемента** (напр. `config: CardConfig` создаётся
  через `new CardConfig()`, т.е. не null) — каркас должен создавать и вложенные
  standalone-объекты, иначе получится невалидный элемент.
- **`members: CardMember[]`** — вложенный массив ссылок ⇒ те же операции применимы
  рекурсивно (`entries.0.members` как контейнер).
- **Порядок и renumber** — новые объекты добавляются в конец и перенумеровываются на save;
  на это опираться нельзя внутри батча (индексы стабилизируются только после `renumber`).
- **Инстансы (collapsed stubs)** — правка массивов внутри source-префаба идёт через сам
  `.prefab`, не через инстанс; на стабе доступны только overrides/mounted (вне рамок этой
  операции).

## 7. Резюме

| Действие | Статус | Причина |
|---|---|---|
| Правка полей существующего элемента (`entries.0.type`, `.config.icon`, `.offsets.0.x`) | ✅ Работает | locateProperty следует по `{__id__}` в value-object (R-навигация) |
| Слияние значений в существующий элемент (`entries.0 = {..}`) | ✅ Безопасно | merge сквозь ref, ссылка сохраняется |
| Добавление нового элемента | ❌ Не поддержано | R1 (листовой writer) + R2 (нет типа/дефолтов); generic-запись ПОРТИТ файл (3a) |
| Удаление элемента по индексу | ❌ Нет операции | нет обёртки splice+guard; сама GC-машинерия есть (R4/F3) |
| Очистка массива (`entries = []`) | ⚠️ Работает | осиротевшие объекты уходят GC, но выборочно так не сделать |

**Вывод:** read→edit для *значений* внутри вложенных CCClass замкнут и надёжен.
Изменение *количества* элементов требует двух новых операций
(`insert_array_element` / `remove_array_element`) плюс guard против footgun 3a; удаление
технически простое (машинерия GC готова), добавление упирается в отсутствие реестра
типов/дефолтов для пользовательских CCClass.
